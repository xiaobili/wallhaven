<objective>
Implement the retry loop that wraps `executeDownload()` and modify the `executeDownload` catch block to support retriable error handling.

**Purpose:** When `executeDownload` fails with a transient error, instead of cleaning up and emitting failure, it should let the error propagate to `executeWithRetry` which manages the retry schedule. The retry loop holds the queue slot via `activeDownloads` persistence (no cleanup during retry). After `MAX_RETRIES` attempts, permanent cleanup and failure notification occur.

**Key design decisions (from 34-CONTEXT.md):**
- D-01: Retry loop lives in main process as `executeWithRetry()` wrapper
- D-04: Slot-holding via activeDownloads persistence — entry stays during retry
- D-06: `executeWithRetry` uses a for loop, not recursion (clean control flow)
- D-10: Modified `executeDownload` catch block skips cleanup for retriable errors
- D-11: Resume downloads also go through retry (offset is passed through)

</objective>

    Modify the `executeDownload` catch block to handle retriable errors differently from permanent errors.

    **Current behavior (for all non-CanceledError):**
    1. Log failure
    2. `cleanupDownload(taskId)` — remove from activeDownloads, delete temp file
    3. Emit 'failed' progress event
    4. Re-throw

    **New behavior — branching on error type:**

    ```
    catch (error: any):
      if CanceledError → throw error (unchanged)
      
      const classification = classifyDownloadError(error)
      
      if classification.isRetriable AND retries remain (retryCount < MAX_RETRIES):
        → Log retriable failure
        → Skip cleanupDownload (holds queue slot per DL-09)
        → Skip 'failed' event (no UI notification during retry per D-12)
        → Throw original error (executeWithRetry catches and retries)
      
      else (permanent error OR max retries exhausted):
        → Log permanent failure (existing behavior)
        → cleanupDownload(taskId) (existing behavior)
        → Emit 'failed' progress event (existing behavior)
        → Throw error (existing behavior)
    ```

    **Important note about `retryCount`:** The catch block checks `activeDownloads.get(taskId)?.retryCount ?? 0 < MAX_RETRIES`. On the first failure, retryCount is undefined (0), so retries are available. After `executeWithRetry` increments retryCount, subsequent executeDownload attempts check the updated value. After MAX_RETRIES attempts, the condition fails and the permanent error path runs.

    Replace the catch block (from `catch (error: any) {` through the closing `}` before `}` that closes the try block):

    ```typescript
    } catch (error: any) {
        // CanceledError — always re-throw without side effects (unchanged)
        if (error.name === 'CanceledError' || error.code === 'ERR_CANCELED') {
          throw error
        }

        // Classify the error
        const classification = classifyDownloadError(error)
        const entry = activeDownloads.get(taskId)

        // Retriable with retries remaining: skip cleanup, let executeWithRetry handle it
        if (
          classification.isRetriable &&
          entry &&
          (entry.retryCount ?? 0) < MAX_RETRIES
        ) {
          logHandler(
            'executeDownload',
            `Retriable error for ${taskId} (attempt ${(entry.retryCount ?? 0) + 1}/${MAX_RETRIES}): ${error.message}`,
            'warn',
          )
          // Skip cleanupDownload — holds queue slot (DL-09)
          // Skip 'failed' progress event — no UI notification during retry (D-12)
          throw error
        }

        // Permanent error or retries exhausted: existing cleanup + emit + throw
        logHandler('executeDownload', `Download failed: ${taskId}: ${error.message}`, 'error')
        cleanupDownload(taskId)

        const windows = BrowserWindow.getAllWindows()
        if (windows.length > 0) {
          windows[0].webContents.send('download-progress', {
            taskId,
            progress: 0,
            offset: 0,
            speed: 0,
            state: 'failed',
            error: error.message || '下载失败',
          })
        }

        throw error
      }
    ```

      for attempt = 1 to MAX_RETRIES:
        try:
          return await executeDownload(taskId, ...)  ← preserves temp file, passes offset
        catch (error):
          if CanceledError → re-throw

          emit log

          // Timer fired → check if task still exists (may have been paused/cancelled)
          if (!activeDownloads.has(taskId)) {
            throw new Error('Download cancelled during retry wait')

          // Continue to next retry attempt
      
      // All retries exhausted:
      log error
      cleanupDownload(taskId)
      emit 'failed' progress with retry-exhausted message
      throw final error

    Implementation:

    ```typescript
    // ---- Retry Loop ----

    /**
     * Execute a download with automatic retry for transient failures.
     * Wraps executeDownload in a retry loop with exponential backoff + full jitter.
     * 
     * - Retriable errors trigger up to MAX_RETRIES (3) retry attempts
     * - Each attempt uses increasing backoff delay (0-2s, 0-4s, 0-8s)
     * - The queue slot is held during retry via activeDownloads persistence (DL-09)
     * - CanceledError and permanent errors pass through immediately
     * - After all retries exhausted: cleanup, emit 'failed', throw
     */
    async function executeWithRetry(
      taskId: string,
      url: string,
      filename: string,
      saveDir: string,
      offset?: number,
    ): Promise<{ filePath: string; size: number }> {
      for (let attempt = 1; attempt <= MAX_RETRIES; attempt++) {
        try {
          // On success, return immediately
          return await executeDownload(taskId, url, filename, saveDir, offset)
        } catch (error: any) {
          // CanceledError — re-throw immediately (pause/cancel during active download)
          if (error.name === 'CanceledError' || error.code === 'ERR_CANCELED') {
            throw error
          }

          // Non-retriable — re-throw immediately
          const classification = classifyDownloadError(error)
          if (!classification.isRetriable) {
            throw error
          }

          // At this point: retriable error, and executeDownload already skipped cleanup.
          // Check if the task still exists (may have been paused/cancelled during the download attempt).
          const entry = activeDownloads.get(taskId)
          if (!entry) {
            throw error
          }

          // Track retry count on the active download entry
          entry.retryCount = attempt

          // Calculate and log backoff
          const delay = calculateBackoff(attempt)
          logHandler(
            'executeWithRetry',
            `Retry ${attempt}/${MAX_RETRIES} for ${taskId} in ${Math.round(delay)}ms: ${error.message}`,
            'info',
          )

          // Wait for backoff duration
          // The timer is stored in retryTimers so PAUSE/CANCEL can cancel it
          await waitWithBackoff(taskId, delay)

          // After wait: verify the task wasn't paused or cancelled during the wait
          if (!activeDownloads.has(taskId)) {
            throw new Error('Download task was paused or cancelled during retry wait')
          }
          // Continue to next retry attempt
        }
      }

      // All MAX_RETRIES attempts failed — permanent failure
      // Get the last error for the failure message
      // (activeDownloads entry still exists because executeDownload skipped cleanup)
      const lastError = new Error('下载失败')
      logHandler('executeWithRetry', `All retries exhausted for ${taskId}`, 'error')

      // Final cleanup — removes from activeDownloads, frees queue slot
      cleanupDownload(taskId)

      // Emit final failure state
      const windows = BrowserWindow.getAllWindows()
      if (windows.length > 0) {
        windows[0].webContents.send('download-progress', {
          taskId,
          progress: 0,
          offset: 0,
          speed: 0,
          state: 'failed',
          error: `下载失败 — 已重试 ${MAX_RETRIES} 次`,
        })
      }

      throw lastError
    }
    ```

    **Edge case: the "last error" message.** At the end of the function, we don't have access to the last thrown error from the loop scope. The `'下载失败 — 已重试 ${MAX_RETRIES} 次'` message is the standardized retry-exhausted message. This is what the user will see in the download list (Phase 35 will format it, but the content is set here).

    **Edge case: what if activeDownloads entry was already cleaned up between the last wait and the exhausted path?** This can't happen in normal flow because:
    - Only PAUSE/CANCEL cleans up the entry
    - PAUSE/CANCEL would cancel the retry timer, which would make `waitWithBackoff` never resolve
    - If the timer does fire (resolve), it means no PAUSE/CANCEL happened
    - After the last attempt fails, we're past the loop and the entry still exists
    - `cleanupDownload(taskId)` safely handles the case where entry doesn't exist (early return)

    **Edge case: what if executeDownload succeeds on retry attempt N but the activeDownloads entry was modified?** executeDownload's success path does `activeDownloads.delete(taskId)` on line ~394, then returns `{ filePath, size }`. executeWithRetry's `return await executeDownload(...)` returns the result. The queue slot is freed because activeDownloads entry no longer exists. This is correct.
