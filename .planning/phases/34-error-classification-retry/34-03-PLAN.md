<objective>
Integrate the retry infrastructure into the download queue's onDequeue callback and the PAUSE/CANCEL handlers.

**Purpose:** Wire everything together so that:
- The queue's onDequeue calls `executeWithRetry` (which handles retries internally) instead of `executeDownload` directly
- PAUSE handler cancels any pending retry timer before attempting to pause a download
- CANCEL handler cancels any pending retry timer before attempting to cancel a download
- `processQueue()` in onDequeue's finally block frees the slot after retries complete (success, exhausted, or cancelled)

**Key decisions (from 34-CONTEXT.md):**
- D-07: PAUSE handler calls cancelRetryTimer at entry, writes state, preserves temp, emits paused
- D-08: CANCEL handler calls cancelRetryTimer at entry, cleans up temp+state, emits cancelled
- D-09 (slot-holding): The slot is held during retry via activeDownloads persistence; PAUSE/CANCEL free the slot explicitly

</objective>

    Before:

    ```typescript
      } finally {
        downloadQueue.processQueue()
      }
    },
    ```

    **How this interacts with slot-holding (DL-09):**

    | State transition | What happens to activeDownloads | processQueue after? |
    |---|---|---|
    | Download succeeds | executeDownload does `activeDownloads.delete(taskId)` | processQueue sees free slot → starts next |
    | Permanent error | executeDownload does `cleanupDownload(taskId)` | processQueue sees free slot → starts next |
    | Retries exhausted | executeWithRetry does `cleanupDownload(taskId)` | processQueue sees free slot → starts next |
    | Paused during backoff | PAUSE handler does `cleanupDownload(taskId, true)` | processQueue called by PAUSE handler + redundant call in finally (harmless) |
    | Cancelled during backoff | CANCEL handler does `cleanupDownload(taskId)` | processQueue called by CANCEL handler + redundant call in finally (harmless) |

    **Why the finally still works:** `processQueue()` reads `activeDownloads.size` fresh from the Map. If the slot was already freed by PAUSE/CANCEL, `processQueue()` sees the correct count and may or may not start a new task. If the slot was freed by executeDownload/executeWithRetry, similarly correct. `processQueue` is idempotent — calling it multiple times is safe.

  <action>
    Add `cancelRetryTimer(taskId)` call to the PAUSE handler **after** the queue-removal check (which handles tasks waiting in queue) and **before** the `activeDownloads.get(taskId)` lookup.

    **Why this order:**
    1. If the task is in the queue (not yet started): `downloadQueue.remove(taskId)` handles it. No retry timer exists because the task never started downloading.
    2. If the task is not in the queue: it's either active downloading OR waiting for retry backoff. In both cases, we need to cancel any pending retry timer.
    3. After cancelRetryTimer, we check activeDownloads and proceed with the existing PAUSE logic.

    The cancelRetryTimer call is **safe to call even when no timer exists** (it checks internally). For active downloads that aren't in retry, calling cancelRetryTimer is a harmless no-op.

    Modified PAUSE handler structure:

    ```typescript
    ipcMain.handle(IPC_CHANNELS.PAUSE_DOWNLOAD_TASK, async (_event, taskId: string) => {
      // 1. If task is waiting in queue, just remove it (no download to pause, no timer to cancel)
      if (downloadQueue.remove(taskId)) {
        return { success: true, state: 'removed_from_queue' }
      }

      // 2. Cancel any pending retry timer (Plan 34 — task may be waiting for backoff)
      cancelRetryTimer(taskId)

      // 3. Must be an active or retry-waiting task
      const download = activeDownloads.get(taskId)
      if (!download) {
        return {
          success: false,
          error: 'Download task not found or already completed',
        }
      }

      // 4. Existing PAUSE logic (unchanged):
      //    abort, write state, emit paused, cleanupDownload(true), processQueue
      try {
        download.abortController.abort()
        // ... rest of existing PAUSE handler unchanged ...
    ```

    The rest of the PAUSE handler body (after `cancelRetryTimer(taskId)`) remains **completely unchanged**. The `cancelRetryTimer` call is the only addition.

    **Edge case — PAUSE during active download:** If the download is actively running (axios request in progress), `download.abortController.abort()` cancels it. The `executeDownload` function's catch block catches CanceledError and re-throws (no cleanup). `executeWithRetry` catches CanceledError and re-throws. The error propagates to the queue's fire-and-forget .catch() handler. Meanwhile, the PAUSE handler continues: writes state file, emits paused, cleanupDownload(true), processQueue.

    **Edge case — PAUSE during retry backoff:** If the download is waiting for retry, no axios request is in flight. `cancelRetryTimer(taskId)` clears the timer. `waitWithBackoff`'s promise never resolves. The `abortController.abort()` is a no-op (no active request). The PAUSE handler writes the state file (from the temp file data that was preserved by executeDownload's retriable-error path), emits paused, cleanupDownload(true), processQueue. The `executeWithRetry` function is still awaiting `waitWithBackoff`, but since the timer was cleared, it stays pending forever. This is fine because the activeDownloads entry has been removed by PAUSE's cleanup. The pending promise is harmless (it will be garbage collected when the queue's internal reference to the task goes out of scope).

    > **Note on the pending promise:** The onDequeue callback's promise never settles in this case (it's stuck on the unresolved waitWithBackoff). This is acceptable for our architecture where the queue is fire-and-forget (`.catch()` only handles errors, no settlement is needed for correct state). The JavaScript runtime's garbage collector will eventually reclaim the promise chain when the queue shifts the task reference away. This matches the pattern used by other fire-and-forget async operations in Electron main process code.
  </action>
