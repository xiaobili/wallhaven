    ```typescript
    // Shared unfavorite logic used by both card badge and ImagePreview handlers
    const unfavoriteWallpaper = async (wallpaperId: string): Promise<void> => {
      if (selectedCollectionId.value) {
        // D-04: specific collection view — remove from this collection only
        // If wallpaper only belongs to this collection, single remove() removes it from all views
        await remove(wallpaperId, selectedCollectionId.value)
      } else {
        // D-05: "all favorites" view — remove from ALL collections
        // Collect all collection IDs first (snapshot) to avoid stale iteration (Pitfall 1)
        const collectionIds = favorites.value
          .filter((f) => f.wallpaperId === wallpaperId)
          .map((f) => f.collectionId)
        for (const cid of collectionIds) {
          await remove(wallpaperId, cid)
        }
      }
    }
    ```

    **Key design decisions:**
    - The "all favorites" case collects collection IDs into a snapshot array BEFORE iteration (per RESEARCH Pitfall 1). This prevents stale data issues since `remove()` calls `load()` internally, which mutates the `favorites` reactive ref.
    - `remove()` calls `showSuccess('已从收藏移除')` internally. For the "all favorites" multi-collection case, N collections produce N success toasts. This is the accepted behavior that maintains View->Composable layering without scope creep.
    - D-04 special case ("如果壁纸只属于该收藏夹，则同时从全部收藏列表中移除") is automatically satisfied: if the wallpaper is only in the current collection, removing from that collection removes it from all views via `load()`.
