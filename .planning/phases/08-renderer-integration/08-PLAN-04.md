# PLAN 04: Integrate Auto-Restore into App Initialization

## Objective

Call `restorePendingDownloads()` during app initialization in `src/main.ts` so pending downloads are automatically restored when the app starts.

## Context

- `src/main.ts` has an `initializeApp()` function that runs after app mount
- Currently loads settings and download history
- Need to add call to `restorePendingDownloads()` from the `useDownload` composable
- Order: load settings first, then restore pending downloads (needs download path from settings)

## Tasks

<name>Add restorePendingDownloads call to initializeApp</name>
<read_first>
- src/main.ts (lines 67-79, initializeApp function)
- src/composables/download/useDownload.ts (restorePendingDownloads method from PLAN 03)

<action>
Modify the `initializeApp` function in `src/main.ts` to add a call to `restorePendingDownloads()`.

Current `initializeApp` function (lines 67-79):
```typescript
async function initializeApp() {
  const { useSettings, useDownload } = await import('./composables')

  await useSettings().load()
  await useDownload().loadHistory()
  console.log('[Main] 应用初始化完成，已从 electron-store 加载数据')

  // 注意：下载进度监听器已移至 useDownload composable 中
  // 这样每个使用下载功能的组件会自行管理监听器的生命周期
}
```

Replace with:
```typescript
async function initializeApp() {
  const { useSettings, useDownload } = await import('./composables')

  await useSettings().load()
  await useDownload().loadHistory()
  await useDownload().restorePendingDownloads()
  console.log('[Main] 应用初始化完成，已从 electron-store 加载数据，并恢复待处理下载任务')

  // 注意：下载进度监听器已移至 useDownload composable 中
  // 这样每个使用下载功能的组件会自行管理监听器的生命周期
}
```
</action>

## Notes

- Settings must be loaded first because `restorePendingDownloads` internally calls `downloadService.getDownloadPath()` which reads the download path from settings
- The order is: `useSettings().load()` → `useDownload().loadHistory()` → `useDownload().restorePendingDownloads()`
