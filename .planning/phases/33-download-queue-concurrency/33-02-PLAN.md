    ```typescript
    ipcMain.handle(
      IPC_CHANNELS.RESUME_DOWNLOAD_TASK,
      async (_event, params: ResumeDownloadParams) => {
        if (!isResumeDownloadParams(params)) {
          return { success: false, error: { code: 'INVALID_PARAMS', message: 'Invalid resume parameters' } }
        }

        const { taskId, url, filename, saveDir, offset } = params
        const tempPath = path.join(saveDir, filename + '.download')

        // 1. Check temp file exists and is valid
        if (!fs.existsSync(tempPath)) {
          return { success: false, error: RESUME_ERRORS.FILE_NOT_FOUND }
        }

        const actualSize = fs.statSync(tempPath).size
        if (actualSize < offset) {
          // Temp file corrupted, clean up
          try { fs.unlinkSync(tempPath) } catch { }

          if (fs.existsSync(statePath)) { try { fs.unlinkSync(statePath) } catch { } }
          return { success: false, error: RESUME_ERRORS.INVALID_OFFSET }

        downloadQueue.enqueue({ taskId, url, filename, saveDir })

        return { success: true, taskId }
      },
    )
    ```

    - But wait -- executeDownload() does a fresh download (no Range header). Resume needs to set up the Range request. So we can't just use the current executeDownload() as-is.

    **IMPORTANT:** This reveals a gap. The current `executeDownload()` does a fresh GET without Range headers. Resume needs to send `Range: bytes=offset-`. The simplest architectural fix: Add an optional `offset` parameter to `executeDownload()`:

    ```typescript
    export async function executeDownload(
      taskId: string,
      url: string,
      filename: string,
      saveDir: string,
      offset?: number,  // NEW: for resume support
    ): Promise<{ filePath: string; size: number }>
    ```

    In `executeDownload()`:
    - If `offset > 0`: set `headers['Range'] = bytes=${offset}-` and use append mode for write stream (`{ flags: 'a' }`)
    - Handle both 206 (continue) and 200 (restart from 0) responses, reusing the resume logic from the current RESUME handler (lines ~637-694)
    - If `offset === 0 || offset === undefined`: current fresh download behavior (no Range header)

    And update the `onDequeue` callback in Plan 01 to pass optional data. Actually, we should add an optional `offset` field to `QueuedDownload`:

    ```typescript
    export interface QueuedDownload {
      taskId: string
      url: string
      filename: string
      saveDir: string
      offset?: number  // NEW: for resume support — 0 or undefined means fresh download
    }
    ```

    Updated onDequeue callback in Plan 01:
    ```typescript
    async (item: QueuedDownload) => {
      try {
        await executeDownload(item.taskId, item.url, item.filename, item.saveDir, item.offset)
      } finally {
        downloadQueue.processQueue()
      }
    }
    ```

    **SPECIFIC INSTRUCTIONS for executeDownload offset support:**
    - Add `offset?: number` to the function signature
    - At the start of the download section (after activeDownloads.set), if `offset > 0`: set `headers['Range'] = 'bytes=' + offset + '-'`
    - Handle response status: if 206, use append mode (`{ flags: 'a' }`) and calculate effectiveTotalSize = offset + contentLength; if 200, delete temp file, use write mode (`{ flags: 'w' }`), reset offset to 0
    - At the completion stage, file rename from tempPath to filePath still works (the temp file already has the partial content + new content appended)

    ADD an import for `fs` if not already at top (it should already be imported):
    ```typescript
    import * as fs from 'fs'
    import * as path from 'path'
    ```

    ```typescript
    downloadQueue.enqueue({ taskId, url, filename, saveDir, offset })
    ```

    This is the complete change. The RESUME handler becomes a validator+enqueuer. The actual resume execution happens inside executeDownload when the queue dequeues it.

    ```typescript
      try {
        const { store } = await import('../../index')
        store.set(key, value)

        // DL-03: Live propagation of maxConcurrentDownloads
        if (key === 'appSettings') {
          const { getQueueInstance } = await import('./download-queue')
          getQueueInstance()?.processQueue()
        }

        return { success: true }
      } catch (error: any) {
        logHandler('store-set', `Error: ${error.message}`, 'error')
        return { success: false, error: error.message }
      }
    })
    ```

    Key design decisions (as documented in CONTEXT.md D-15 and RESEARCH.md Open Question #3):
    - Dynamic import is used (not static) to avoid any circular dependency since store.handler.ts uses `import('../../index')` already
    - `getQueueInstance()?.processQueue()` uses optional chaining because the queue might not be initialized yet (Plan 01 might not have run)
    - This triggers re-evaluation when the setting changes BUT does NOT interrupt active downloads (DL-04) because `processQueue()` only starts new tasks when `activeDownloads.size < maxConcurrentDownloads`
    - When user lowers the limit (e.g., 5 to 2): `processQueue()` reads the new limit (2), checks `activeDownloads.size` (which might be 5), and starts NO new tasks because 5 >= 2. Active tasks complete normally. DL-04 is satisfied implicitly.
    - When user raises the limit (e.g., 3 to 5): `processQueue()` reads the new limit (5), checks `activeDownloads.size` (which might be 3), and starts up to 2 more waiting tasks. DL-03 is satisfied.
