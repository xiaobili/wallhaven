| ID | 需求描述 | 复杂度 |

| ARCH-02 | 创建 `src/shared/types/ipc.ts`，定义所有 IPC 通道类型 | 中 |
| ARCH-03 | 创建 `src/errors/` 错误类定义，统一错误处理 | 低 |
| ARCH-04 | 创建 `useAlert` composable，解决 Alert 逻辑重复问题 | 低 |
| ARCH-05 | 添加全局错误处理器，处理未捕获的 Promise rejection | 低 |
| ARCH-06 | 消除 Store 中的 `any` 类型（actions.ts 等） | 低 |

### 成功标准

1. **编译通过**：TypeScript 编译无错误
2. **功能不变**：所有现有功能正常运行

4. **类型安全**：Store 中无 `any` 类型

| 风险 | 等级 | 缓解措施 |

## 阶段 2：数据层抽象

### 目标

解耦 Store 与 Electron API 的直接依赖，建立 Repository 模式。

### 需求映射

| ID | 需求描述 | 复杂度 |
|----|----------|--------|
| DATA-01 | 创建 `ElectronClient`，封装 `window.electronAPI` 调用 | 中 |
| DATA-02 | 创建 `ApiClient`，封装 HTTP 请求逻辑 | 中 |
| DATA-03 | 创建 `StoreClient`，封装持久化存储操作 | 中 |
| DATA-04 | 创建 `SettingsRepository`，抽象设置数据访问 | 低 |
| DATA-05 | 创建 `DownloadRepository`，抽象下载数据访问 | 低 |
| DATA-06 | 创建 `WallpaperRepository`，抽象壁纸数据访问 | 低 |

### 成功标准

1. **可测试性**：Repository 层可独立 mock 和测试
2. **解耦完成**：Store 通过 Repository 访问数据，不直接调用 Electron API
3. **功能不变**：所有现有功能正常运行
4. **代码质量**：每个 Client/Repository 文件 < 150 行

### 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| 抽象层引入 bug | 中 | 保持简单封装，不改变原有逻辑 |
| 持久化数据丢失 | 高 | 保持存储键名不变，向后兼容 |

### 依赖关系

依赖阶段 1 完成的类型定义和错误处理机制。

## 阶段 3：业务层与组合层

### 目标

实现业务逻辑抽象，简化 Store，建立清晰的分层架构。

### 需求映射

| ID | 需求描述 | 复杂度 |
|----|----------|--------|
| BIZ-01 | 创建 `WallpaperService`，实现壁纸业务逻辑 | 中 |
| BIZ-02 | 创建 `DownloadService`，实现下载业务逻辑 | 中 |
| BIZ-03 | 创建 `SettingsService`，实现设置业务逻辑 | 低 |
| BIZ-04 | 创建 `useWallpaperList` composable | 中 |
| BIZ-05 | 创建 `useDownload` composable | 中 |
| BIZ-06 | 创建 `useSettings` composable | 低 |
| BIZ-07 | 重构 Store，移除业务逻辑，仅保留响应式状态 | 高 |

### 成功标准

1. **职责清晰**：Store 仅存储响应式状态，无业务逻辑
2. **分层明确**：业务逻辑集中在 Service 层
3. **可复用性**：Composable 协调 Service 和 Store，组件逻辑简化
4. **功能不变**：所有现有功能正常运行

### 风险与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|----------|
| Store 重构破坏响应式 | 高 | 保持 reactive/ref 使用方式不变 |
| 业务逻辑遗漏 | 中 | 逐一对照现有实现，确保逻辑完整 |

### 依赖关系

依赖阶段 2 的 Repository 层。
