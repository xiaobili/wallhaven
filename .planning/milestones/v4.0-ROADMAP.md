## Key Decisions

- DownloadQueue lives in main process (download.handler.ts), not renderer
- Queue uses callback injection instead of direct import — avoids circular dependency
- Singleton accessors (setQueueInstance/getQueueInstance) — enables cross-module queue access
- Main process is source of truth for download state transitions
- executeDownload() handles both fresh and resume via optional offset parameter
- Dynamic import for getQueueInstance in store.handler.ts prevents circular dependency
- Retry holds queue slot during backoff — prevents starvation
- classifyDownloadError() maps error codes/statuses to retriable/permanent
- Backoff uses full jitter (random 0 to exponential cap) — prevents thundering herd
- waitWithBackoff stores timer + resolve callback in retryTimers Map — enables PAUSE/CANCEL
- executeWithRetry checks activeDownloads.has(taskId) after wait — detects pause/cancel
- executeDownload catch block skips cleanup/emit for retriable errors when retries remain
- Null/undefined error guard in executeDownload catch — prevents permanent slot blockage
- New DownloadState 'retrying' added to state union; new fields retryCount, retryDelay
